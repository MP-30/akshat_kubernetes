EKS Lab · Kubernetes Ingress

Host-Based Ingress Routing
on Amazon EKS

Complete end-to-end lab — deploy apps, install NGINX Ingress Controller, configure host-based routing using nip.io, and clean up.

Platform: Amazon EKS ap-south-1
Ingress: nginx v1.10.1
Routing: HTTP Host header
DNS: nip.io wildcard

// Architecture

What We Are Building

Three sample apps + a default backend, each exposed under a unique hostname. One NGINX Ingress Controller reads the HTTP Host header and routes to the correct backend — all through a single NodePort.

02
Install NGINX Ingress Controller
deploy.yaml from kubernetes community — creates controller in ingress-nginx namespace

This single command deploys the NGINX Ingress Controller, its ServiceAccount, ClusterRole, and a Service of type LoadBalancer in the ingress-nginx namespace.

bash — install controller
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/controller-v1.10.1/deploy/static/provider/aws/deploy.yaml

# Wait until the controller pod is Running
kubectl get pods -n ingress-nginx -w
⚠ On AWS Academy accounts, LoadBalancer creation is blocked. The ingress-nginx-controller service will show EXTERNAL-IP <pending> permanently. Patch it to NodePort as shown below.
bash — patch to NodePort (Academy accounts)
kubectl patch svc ingress-nginx-controller \
  -n ingress-nginx \
  -p '{"spec": {"type": "NodePort"}}'
bash — get NodePort number
kubectl get svc -n ingress-nginx ingress-nginx-controller

# Look for:  80:XXXXX/TCP  — that XXXXX is your NodePort
# Example:   80:32606/TCP  → NodePort is 32606
bash — get worker node public IP
kubectl get nodes -o wide

# Look at the EXTERNAL-IP column
# Example output:
# NAME                                           EXTERNAL-IP
# ip-192-168-7-124.ap-south-1.compute.internal   65.2.186.80   ← use this
# ip-192-168-94-191.ap-south-1.compute.internal  43.205.93.11
ℹ You have 2 worker nodes. Pick either public IP — NodePort is cluster-wide so traffic hitting any node gets routed to the correct pod automatically.

Open the NodePort in the EC2 Security Group for both worker nodes:

bash — open NodePort in Security Group
# Get the Security Group IDs of both worker nodes
aws ec2 describe-instances \
  --filters "Name=private-ip-address,Values=192.168.7.124" \
  --query "Reservations[].Instances[].SecurityGroups" \
  --region ap-south-1

# Add inbound rule (replace sg-xxxxxxxxx with actual SG ID)
aws ec2 authorize-security-group-ingress \
  --group-id sg-xxxxxxxxx \
  --protocol tcp \
  --port 32606 \
  --cidr 0.0.0.0/0 \
  --region ap-south-1

# Repeat for the second worker node SG
03
Create the Host-Based Ingress
ingress-host.yaml — the host: field per rule is what makes this host-based

Each rule has a host: field. NGINX reads the HTTP Host header from every incoming request and matches it against these hostnames to decide which backend to forward to.

ℹ nip.io is a free public DNS service. Any hostname like anything.YOUR_IP.nip.io automatically resolves to YOUR_IP. No DNS configuration needed — just use the URL directly in your browser.
ingress-host.yaml
yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: apps-host-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:

    # ── Default backend ──────────────────────────────────────
  - host: "default.65.2.186.80.nip.io"
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: nginx-custom-default-backend
            port:
              number: 80

    # ── pyapp ─────────────────────────────────────────────────
  - host: "pyapp.65.2.186.80.nip.io"
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: pyapp-svc
            port:
              number: 80

    # ── unapp ─────────────────────────────────────────────────
  - host: "unapp.65.2.186.80.nip.io"
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: unapp-svc
            port:
              number: 80

    # ── petclinic ─────────────────────────────────────────────
  - host: "petclinic.65.2.186.80.nip.io"
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: petclinic-svc
            port:
              number: 80
bash — apply
kubectl apply -f ingress-host.yaml

# Verify — HOSTS column should show all four nip.io hostnames
kubectl get ingress apps-host-ingress

Open in browser — always include the NodePort in the URL

05
Delete All Resources
cleanup.yaml — single file to delete everything created in this lab

You can delete everything using the same files you used to create them. kubectl delete -f reads the yaml and removes all matching resources.

bash — delete all (recommended)
# 1. Delete the Ingress rule
kubectl delete -f ingress-host.yaml

# 2. Delete all app deployments and services
kubectl delete -f apps.yaml

# 3. Delete the NGINX Ingress Controller (entire ingress-nginx namespace)
kubectl delete -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/controller-v1.10.1/deploy/static/provider/aws/deploy.yaml

Alternatively, create a single cleanup.yaml that deletes everything using kubectl delete in the right order:
