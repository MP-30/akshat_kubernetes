Case Study 01 Host-Based Routing EKS Setup
A startup wants to host three apps on one EKS cluster — each on its own subdomain
// Scenario

You are a DevOps engineer at a startup. The team has three applications ready to deploy on an EKS cluster in ap-south-1:

App 1
frontend
port 3000
App 2
api-server
port 8080
App 3
admin-panel
port 80
Worker Node IP
13.235.10.45

The team wants users to access each app at its own subdomain — not via paths like /frontend. The account is an AWS Academy account. Services and Deployments are already created.

// Your Tasks
  • T1What type of Ingress routing should be used — path-based or host-based? Why?
  • T2Write the complete ingress.yaml using nip.io hostnames for all three apps.
  • T3The NGINX Ingress Controller service is stuck at EXTERNAL-IP <pending>. What do you do?
  • T4After patching to NodePort, the NodePort assigned is 31080. What URLs do users visit?
// T1 — Routing Type

Host-based routing is the right choice here. The requirement is one subdomain per app (frontend.domain.com, api.domain.com) — not URL paths. Path-based would give domain.com/frontend which is not what the team wants.

Rule of thumb: If apps need their own domain/subdomain → host-based. If apps share one domain but differ by URL path → path-based.
// T2 — ingress.yaml

Each rule gets a host: field with the nip.io hostname. All services use port 80 on the service side (services map internal ports to 80 externally).

yaml — ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: startup-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:

  - host: "frontend.13.235.10.45.nip.io"
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend-svc
            port:
              number: 80

  - host: "api.13.235.10.45.nip.io"
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: api-server-svc
            port:
              number: 80

  - host: "admin.13.235.10.45.nip.io"
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: admin-panel-svc
            port:
              number: 80
// T3 — Fix EXTERNAL-IP Pending

AWS Academy accounts cannot create ELBs. The fix is to patch the Ingress Controller service from LoadBalancer to NodePort:

bash
kubectl patch svc ingress-nginx-controller \
  -n ingress-nginx \
  -p '{"spec": {"type": "NodePort"}}'

# Confirm it's changed and note the NodePort
kubectl get svc -n ingress-nginx ingress-nginx-controller

Also open the assigned NodePort in the EC2 Security Group of both worker nodes (inbound TCP rule).

// T4 — Final Access URLs

With NodePort 31080, users access apps at:

AppURL
frontendhttp://frontend.13.235.10.45.nip.io:31080
api-serverhttp://api.13.235.10.45.nip.io:31080
admin-panelhttp://admin.13.235.10.45.nip.io:31080
Always include the NodePort in the URL. Without it, the browser defaults to port 80 and gets a connection timeout — the controller is not on port 80.
Case Study 02 Debugging Find the Bug
A developer deployed an Ingress but every URL returns 404 or the wrong app
// Scenario

A junior developer on your team set up host-based Ingress routing for two apps — blogapp and shopapp — on an EKS cluster. Worker node public IP is 52.66.18.90, NodePort is 30500.

The developer says: "I visit blogapp.52.66.18.90.nip.io:30500 and get 404. When I visit shopapp.52.66.18.90.nip.io:30500 it loads blogapp instead of shopapp."

Here is the Ingress YAML the developer wrote. Find all the mistakes:

yaml — buggy ingress (find the mistakes)
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: blog-shop-ingress
  annotations:
    ingress.kubernetes.io/rewrite-target: /   ← Bug 1
spec:
  ingressClassName: nginx
  rules:
  - host: blogapp.52.66.18.90              ← Bug 2
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: blog-svc
            port:
              number: 80
  - http:                                    ← Bug 3
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: shop-svc
            port:
              number: 80
// Your Tasks
  • T1Identify all three bugs in the YAML above and explain what effect each one causes.
  • T2Write the corrected ingress.yaml with all bugs fixed.
  • T3What command would you run to verify the fix was applied and both hosts are registered?
// T1 — The Three Bugs
  • ✕
    Bug 1 — Wrong annotation prefix
    ingress.kubernetes.io/rewrite-target is silently ignored by the NGINX controller. The correct prefix is nginx.ingress.kubernetes.io/rewrite-target. Effect: rewrite never happens — apps may receive wrong path prefixes.
  • ✕
    Bug 2 — Missing .nip.io suffix
    host: blogapp.52.66.18.90 is not a valid hostname that any DNS can resolve. It should be blogapp.52.66.18.90.nip.io. Effect: the browser sends the correct Host header but NGINX finds no matching rule → 404.
  • ✕
    Bug 3 — Missing host: field for shopapp
    The second rule has no host: field at all. A rule without a host matches ALL incoming requests regardless of hostname. Effect: any request that doesn't match the first rule falls through to shopapp — including requests meant for blogapp. This is why blogapp.nip.io also serves shopapp.
// T2 — Corrected ingress.yaml
yaml — fixed
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: blog-shop-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /   ✓ correct prefix
spec:
  ingressClassName: nginx
  rules:
  - host: "blogapp.52.66.18.90.nip.io"           ✓ full nip.io hostname
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: blog-svc
            port:
              number: 80
  - host: "shopapp.52.66.18.90.nip.io"           ✓ host: field added
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: shop-svc
            port:
              number: 80
// T3 — Verify the Fix
bash
# Apply the corrected file
kubectl apply -f ingress.yaml

# Check HOSTS column — both hostnames should appear
kubectl get ingress blog-shop-ingress

# Test each route manually using curl
curl -I http://blogapp.52.66.18.90.nip.io:30500
curl -I http://shopapp.52.66.18.90.nip.io:30500

# Or test by sending the Host header directly to the IP
curl -v http://52.66.18.90:30500 -H "Host: blogapp.52.66.18.90.nip.io"
curl -v http://52.66.18.90:30500 -H "Host: shopapp.52.66.18.90.nip.io"
Expected result: Both curls return HTTP 200. The kubectl get ingress HOSTS column shows both blogapp.52.66.18.90.nip.io and shopapp.52.66.18.90.nip.io.
Case Study 03 Design Question Mixed Routing
Design an Ingress for an e-commerce platform with both host-based and path-based rules
// Scenario

You are setting up Ingress for an e-commerce company's EKS cluster. The platform has the following requirements from the product team:

Main store
store-svc
shop.mystore.com → /
API v1
api-v1-svc
api.mystore.com/v1
API v2
api-v2-svc
api.mystore.com/v2
Worker Node IP
3.110.45.22

Use nip.io in place of real DNS. The API has two versions running as separate deployments — both should be accessible under the same api.* hostname but different paths.

// Your Tasks
  • T1Explain the routing design — which rules use host-based and which use path-based routing?
  • T2Write the complete ingress.yaml implementing all three routing rules.
  • T3What happens if a user visits api.3.110.45.22.nip.io:NodePort/v3 — a path that has no rule? How would you handle this?
// T1 — Routing Design

This scenario uses both types of routing together:

RuleHostnamePathTypeBackend
1shop.3.110.45.22.nip.io/Host-basedstore-svc
2api.3.110.45.22.nip.io/v1Host + Pathapi-v1-svc
3api.3.110.45.22.nip.io/v2Host + Pathapi-v2-svc

Rules 2 and 3 share the same host: value but differ by path. They are written as two entries under the same host: block. NGINX first matches the hostname, then matches the longer/more specific path.

// T2 — ingress.yaml
yaml — ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: ecommerce-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:

    # ── Main store (host-based only) ──────────────────────────
  - host: "shop.3.110.45.22.nip.io"
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: store-svc
            port:
              number: 80

    # ── API (host + path — both versions same hostname) ───────
  - host: "api.3.110.45.22.nip.io"
    http:
      paths:
      - path: /v1           # api.nip.io/v1 → api-v1-svc
        pathType: Prefix
        backend:
          service:
            name: api-v1-svc
            port:
              number: 80
      - path: /v2           # api.nip.io/v2 → api-v2-svc
        pathType: Prefix
        backend:
          service:
            name: api-v2-svc
            port:
              number: 80
// T3 — Handling Unmatched Paths (/v3)

A request to api.nip.io/v3 matches the api.* hostname but no path rule exists for /v3. NGINX returns 404 Not Found.

There are two clean ways to handle this:

  • Option A — Add a catch-all path: Add a path: / rule at the end of the api host block pointing to a fallback or error service. Since NGINX prefers longer paths, /v1 and /v2 still win for their respective prefixes.
  • Option B — Use defaultBackend: Configure a global defaultBackend in the Ingress spec to catch anything that doesn't match any host or path rule.
yaml — Option A: catch-all path for unknown api versions
  - host: "api.3.110.45.22.nip.io"
    http:
      paths:
      - path: /v1
        pathType: Prefix
        backend:
          service: {name: api-v1-svc, port: {number: 80}}
      - path: /v2
        pathType: Prefix
        backend:
          service: {name: api-v2-svc, port: {number: 80}}
      - path: /               # catch-all — /v3, /v4, anything unknown
        pathType: Prefix
        backend:
          service: {name: api-v1-svc, port: {number: 80}}
Path priority: NGINX always picks the longest matching path. /v2 beats / for requests starting with /v2. The catch-all / only fires when nothing more specific matches.
