Concepts
Q1 What is a Kubernetes Ingress and why do we need it?

Ingress is a Kubernetes API object that manages external HTTP/HTTPS access to services inside the cluster. It acts as a smart entry point that routes incoming requests to the correct backend service based on rules you define.

Without Ingress, you need one LoadBalancer Service per application — which means one AWS ELB per app, which is expensive and hard to manage. Ingress consolidates all traffic through a single entry point.

Without IngressWith Ingress
3 apps = 3 LoadBalancers3 apps = 1 LoadBalancer
3 different IPs for users1 IP, rules differentiate apps
TLS configured per serviceTLS terminated once at Ingress
Q2 What is the difference between an Ingress and an Ingress Controller?

Ingress is just a YAML configuration object — it defines the routing rules (which hostname/path goes to which service). By itself it does nothing.

Ingress Controller is the actual software (a pod running in your cluster) that reads those Ingress rules and implements them. It watches for Ingress objects and configures itself to route traffic accordingly.

Analogy: Ingress is the traffic rulebook. Ingress Controller is the traffic cop that reads and enforces the rulebook.

Common Ingress Controllers: NGINX, Traefik, AWS ALB Ingress Controller, HAProxy. You must install a controller — Kubernetes does not ship one by default.

Q3 What is ingressClassName and why is it needed?

ingressClassName tells Kubernetes which Ingress Controller should handle this Ingress object. A cluster can have multiple Ingress Controllers installed simultaneously (e.g. nginx AND traefik).

yaml
spec:
  ingressClassName: nginx   # this Ingress is handled by the nginx controller

Without it, no controller picks up the Ingress and your routing rules are never applied. Each Ingress Controller registers itself with a class name when installed.

Q4 What is the difference between pathType: Prefix and pathType: Exact?
pathTypeBehaviourExample path: /app
PrefixMatches /app and anything starting with /app/app, /app/home, /app/v1/users all match
ExactMatches only that exact path, nothing elseOnly /app matches. /app/home does NOT
ImplementationSpecificController decides the matching logicDepends on the Ingress Controller used

In most production setups Prefix is used so the entire application hierarchy is covered by one rule.

Q5 What does the rewrite-target annotation do?

When using path-based routing, the path prefix needs to be stripped before forwarding to the backend. For example, if a user hits /pyapp/home, the app probably expects just /home — not /pyapp/home.

rewrite-target: / rewrites every incoming path to / before forwarding.

yaml
annotations:
  nginx.ingress.kubernetes.io/rewrite-target: /
Important: The annotation prefix must be nginx.ingress.kubernetes.io/ — using just ingress.kubernetes.io/ is silently ignored by the NGINX controller.

In host-based routing this is usually not needed since all requests already hit /.

Configuration
Q6 Write a minimal Ingress YAML that routes /api to a service called api-svc on port 80.
yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: api-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:
  - http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-svc
            port:
              number: 80
Q7 How do you configure host-based routing in an Ingress? What field do you add?

Add the host: field to each rule. NGINX reads the incoming HTTP Host header and matches it against these values.

yaml
spec:
  rules:
  - host: "app1.example.com"   # ← this is the only change from path-based
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: app1-svc
            port:
              number: 80
  - host: "app2.example.com"
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: app2-svc
            port:
              number: 80

Each hostname must resolve to the Ingress Controller's IP via DNS. In labs, nip.io is used so no DNS setup is needed.

Q8 What happens if a request arrives with a Host header that does not match any Ingress rule?

By default, NGINX returns a 404 Not Found response. To handle unmatched requests gracefully, you can configure a defaultBackend in the Ingress spec:

yaml
spec:
  ingressClassName: nginx
  defaultBackend:          # catches everything that doesn't match
    service:
      name: my-default-backend
      port:
        number: 80
  rules:
      # ... your host rules here
Q9 Can you have both host-based and path-based routing in the same Ingress rule?

Yes. You can specify both a host: and multiple paths within the same rule. NGINX first matches the hostname, then evaluates the paths within that host.

yaml
- host: "api.example.com"
  http:
    paths:
    - path: /v1      # api.example.com/v1 → v1-svc
      pathType: Prefix
      backend:
        service:
          name: v1-svc
          port: {number: 80}
    - path: /v2      # api.example.com/v2 → v2-svc
      pathType: Prefix
      backend:
        service:
          name: v2-svc
          port: {number: 80}

More specific (longer) paths take priority over shorter ones when multiple paths could match.

Q10 What namespace does the NGINX Ingress Controller install into, and why does that matter?

The NGINX Ingress Controller installs into the ingress-nginx namespace. This matters for several reasons:

  • All kubectl commands to check or manage the controller need -n ingress-nginx
  • The controller Service (NodePort/LoadBalancer) is in this namespace — this is where you get the external IP or NodePort
  • Logs are accessed via kubectl logs -n ingress-nginx deployment/ingress-nginx-controller
  • Deleting the controller means deleting from this namespace

The Ingress objects themselves live in the default namespace (or wherever your app is). The controller watches all namespaces.

Routing
Q11 How does NGINX Ingress actually decide which backend to send a request to?

NGINX Ingress evaluates every incoming request using this order of priority:

  • 1. Host header match — finds rules where the host: field matches the HTTP Host header
  • 2. Path match — within the matched host, finds the longest matching path
  • 3. Default backend — if nothing matches, uses defaultBackend if configured, otherwise returns 404
Longer/more specific paths always win over shorter ones. /app/v2 beats /app beats /.
Q12 What is nip.io and why is it used in labs instead of real DNS?

nip.io is a free public wildcard DNS service. Any hostname in the format anything.IP_ADDRESS.nip.io automatically resolves to that IP address — no DNS record creation needed.

HostnameResolves to
pyapp.65.2.186.80.nip.io65.2.186.80
petclinic.65.2.186.80.nip.io65.2.186.80
anything.65.2.186.80.nip.io65.2.186.80

Each hostname resolves to the same IP but sends a different Host: header — which is exactly what NGINX needs to make routing decisions. In production, you'd use real DNS A records instead.

Q13 If you have two worker nodes, which node's IP do you use in the Ingress host rule?

Either one. NodePort is a cluster-wide feature — the port is opened on every worker node. When traffic hits any node on the NodePort, Kubernetes internally routes it to the correct pod regardless of which node the pod is running on.

So pyapp.65.2.186.80.nip.io:32606 and pyapp.43.205.93.11.nip.io:32606 would both work. The important thing is that the hostname you put in the Ingress rule matches the hostname the browser sends in the Host header.

Tip: Pick one IP and use it consistently for all your hostnames in one Ingress file. Don't mix IPs across rules unnecessarily.
Q14 What is the difference between a ClusterIP service and a NodePort service in the context of Ingress?

The backend services referenced in Ingress rules (pyapp-svc, unapp-svc etc.) should be ClusterIP — they only need to be reachable inside the cluster, not from outside. NGINX routes traffic to them internally.

The Ingress Controller's own service needs to be LoadBalancer or NodePort so it can receive traffic from outside the cluster.

ServiceTypeWhy
pyapp-svc, unapp-svc, petclinic-svcClusterIPOnly NGINX needs to reach them
ingress-nginx-controllerLoadBalancer / NodePortMust be reachable from internet
Q15 How do you find the NodePort assigned to the NGINX Ingress Controller?
bash
kubectl get svc -n ingress-nginx ingress-nginx-controller

The output shows something like 80:32606/TCP, 443:31750/TCP. The format is clusterPort:nodePort/protocol. So 32606 is the NodePort for HTTP traffic.

Kubernetes auto-assigns a port from the range 30000–32767. If the cluster is recreated, the NodePort changes — always verify before accessing from the browser.

EKS / AWS
Q16 Why does the Ingress Controller service stay in <pending> on AWS Academy accounts?

The NGINX Ingress Controller creates a Service of type LoadBalancer. On EKS, this triggers AWS to create an Elastic Load Balancer. AWS Academy/Learner Lab accounts have this service restricted.

The exact error from kubectl describe svc ingress-nginx-controller -n ingress-nginx:

error output
OperationNotPermitted: This AWS account currently does not support
creating load balancers. Please contact AWS Support.

The fix is to patch the service to NodePort instead:

bash
kubectl patch svc ingress-nginx-controller \
  -n ingress-nginx \
  -p '{"spec": {"type": "NodePort"}}'
Q17 What Security Group change is needed for NodePort access on EKS?

By default, EC2 Security Groups block all inbound traffic. To access your app through the NodePort, you must add an inbound rule on the worker node's Security Group:

  • Protocol: TCP
  • Port: your NodePort (e.g. 32606)
  • Source: 0.0.0.0/0 (or restrict to your IP for security)

Without this rule, the browser shows ERR_CONNECTION_TIMED_OUT — the connection drops silently at the network level before even reaching Kubernetes.

bash
aws ec2 authorize-security-group-ingress \
  --group-id sg-xxxxxxxxx \
  --protocol tcp \
  --port 32606 \
  --cidr 0.0.0.0/0 \
  --region ap-south-1
Q18 How do you find the public IP of an EKS worker node?

The simplest way on EKS is:

bash
kubectl get nodes -o wide
# Look at the EXTERNAL-IP column

You can also use AWS CLI:

bash
aws ec2 describe-instances \
  --filters "Name=private-ip-address,Values=192.168.7.124" \
  --query "Reservations[].Instances[].PublicIpAddress" \
  --region ap-south-1

Or extract it programmatically:

bash
kubectl get nodes -o jsonpath='{.items[0].status.addresses[?(@.type=="ExternalIP")].address}'

Troubleshooting
Q20 The browser shows ERR_CONNECTION_TIMED_OUT. What are the likely causes?

Connection timeout means the packet never reached the server (or was dropped silently). Check in this order:

  • NodePort missing in URL — browser used port 80 by default, controller is on 32606
  • Security Group blocked — EC2 inbound rule for the NodePort is missing
  • Wrong IP — the IP in the URL doesn't belong to any worker node
  • Controller not running — check kubectl get pods -n ingress-nginx

The fix in 90% of lab cases: add :32606 to the URL, or open the NodePort in the EC2 Security Group.

Q21 NGINX returns 404 even though the Ingress rule is applied. How do you debug this?

A 404 from NGINX means the controller is reachable but no rule matched. Debug steps:

bash
# 1. Check what hosts the Ingress knows about
kubectl get ingress
kubectl describe ingress apps-host-ingress

# 2. Test by sending the Host header manually
curl -v http://65.2.186.80:32606 -H "Host: pyapp.65.2.186.80.nip.io"

# 3. Check NGINX controller logs for routing decisions
kubectl logs -n ingress-nginx deployment/ingress-nginx-controller --tail=30

Common causes: typo in the host: field, wrong annotation prefix (ingress.kubernetes.io/ instead of nginx.ingress.kubernetes.io/), or Ingress not yet synced.

Q22 All hostnames route to the same application. What is the most likely cause?

The host: field is missing from the Ingress rules. Without it, every rule is treated as a path-based rule with no hostname restriction — all requests fall through to the same backend.

yaml
# WRONG — no host field, all traffic goes to pyapp-svc
- http:
    paths:
    - path: /
      backend:
        service:
          name: pyapp-svc
          port: {number: 80}

# CORRECT — host field added
- host: "pyapp.65.2.186.80.nip.io"
  http:
    paths:
    - path: /
      backend:
        service:
          name: pyapp-svc
          port: {number: 80}
Q23 The Ingress rule is applied but kubectl get ingress shows no ADDRESS. What does this mean?

The Ingress Controller is not running or is not watching this Ingress object. Possible reasons:

  • The Ingress Controller was never installed
  • ingressClassName: nginx doesn't match the installed controller's class
  • The controller pod is in CrashLoopBackOff or Pending state
bash — check
# Is the controller running?
kubectl get pods -n ingress-nginx

# What IngressClass names are available?
kubectl get ingressclass

The ADDRESS column populates only after the controller picks up the Ingress and the service has an external IP or NodePort assigned.

Q24 How do you verify that NGINX is actually reading the Host header correctly?

Use curl to manually set the Host header and connect to the raw IP — this is exactly what the browser does when you type the hostname:

bash
# Test pyapp routing
curl -v http://65.2.186.80:32606 -H "Host: pyapp.65.2.186.80.nip.io"

# Test petclinic routing
curl -v http://65.2.186.80:32606 -H "Host: petclinic.65.2.186.80.nip.io"

# Test unmatched host — should return 404
curl -v http://65.2.186.80:32606 -H "Host: unknown.example.com"

This technique isolates the DNS layer completely — if curl with a Host header works but the browser doesn't, the issue is DNS resolution, not Ingress routing.

Q25 A backend returns 503 Service Unavailable through Ingress. What do you check?

503 means NGINX reached the controller but could not forward to the backend. Check in this order:

bash
# 1. Are the backend pods Running?
kubectl get pods

# 2. Does the Service selector match the pod labels?
kubectl describe svc pyapp-svc
kubectl get pods --show-labels

# 3. Are there endpoints registered? (empty = selector mismatch)
kubectl get endpoints pyapp-svc

# 4. Does the Service port match the containerPort?
kubectl describe svc pyapp-svc

The most common cause is a label mismatch between the Service selector and the pod labels — the Service exists but has zero endpoints, so NGINX has nowhere to send traffic.
