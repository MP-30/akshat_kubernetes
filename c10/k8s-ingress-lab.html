02
Install NGINX Ingress Controller
Installs the controller that reads Ingress rules and routes traffic

Without an Ingress Controller, the Ingress resource is just a config object — nothing actually routes traffic. On EKS we use the official NGINX Ingress Controller from the Kubernetes community.

ℹ This single command deploys the controller pods, ClusterRole, ServiceAccount, and a Service of type LoadBalancer in the ingress-nginx namespace.
bash 
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/controller-v1.10.1/deploy/static/provider/aws/deploy.yaml
bash — verify 
# Wait for the controller pod to be Running
kubectl get pods -n ingress-nginx

# Check the service — EXTERNAL-IP will show pending on Academy accounts
kubectl get svc -n ingress-nginx
03
Create the Ingress Resource — app-nginx.yaml
Define path-based routing rules for all four services

The Ingress object tells NGINX which backend service to forward requests to based on the URL path. It uses ingressClassName: nginx to bind to our controller.

⚠ Annotation fix required: Use nginx.ingress.kubernetes.io/rewrite-target — NOT ingress.kubernetes.io/rewrite-target. The wrong prefix is silently ignored.
yaml — app-nginx.yaml 
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: apps-ingress-rule
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:
  - http:
      paths:
        - path: /
          pathType: Prefix
          backend:
            service:
              name: nginx-custom-default-backend
              port:
                number: 80
        - path: /unapp
          pathType: Prefix
          backend:
            service:
              name: unapp-svc
              port:
                number: 80
        - path: /pyapp
          pathType: Prefix
          backend:
            service:
              name: pyapp-svc
              port:
                number: 80
        - path: /petclinic
          pathType: Prefix
          backend:
            service:
              name: petclinic-svc
              port:
                number: 80
bash — apply 
kubectl apply -f app-nginx.yaml
kubectl get ingress
04
Problem: EXTERNAL-IP stays <pending>
AWS Academy accounts cannot create Load Balancers

After installing the controller, you'll notice the ingress-nginx-controller service stays in <pending> state for EXTERNAL-IP. Checking the events reveals the root cause:

✕
Error from kubectl describe svc:

OperationNotPermitted: This AWS account currently does not support creating load balancers.

This is an AWS account-level restriction, not an IAM or Kubernetes problem. AWS Academy / Learner Lab / Sandbox accounts have ELB service disabled. No IAM policy fix will resolve this.

bash — diagnose 
# Check svc status
kubectl get svc -n ingress-nginx

# See the actual AWS error
kubectl describe svc ingress-nginx-controller -n ingress-nginx | tail -20
05
Workaround: Switch to NodePort
Access the Ingress Controller directly via EC2 worker node public IP

Since we can't create an ELB, we patch the controller's service from LoadBalancer to NodePort. Traffic then enters via the worker node's public IP on the auto-assigned NodePort.

✓ NodePort range is 30000–32767. The port 32606 is already assigned to HTTP — check your cluster's actual port with kubectl get svc.

Step 5a — Patch the service to NodePort:

bash 
kubectl patch svc ingress-nginx-controller \
  -n ingress-nginx \
  -p '{"spec": {"type": "NodePort"}}'

Step 5b — Note the assigned NodePort:

bash 
kubectl get svc -n ingress-nginx ingress-nginx-controller
# Look for: 80:XXXXX/TCP  ← that XXXXX is your NodePort

Step 5c — Get the worker node public IP:

bash 
aws ec2 describe-instances \
  --filters "Name=private-ip-address,Values=192.168.7.124" \
  --query "Reservations[].Instances[].PublicIpAddress" \
  --region ap-south-1

Step 5d — Open the NodePort in the EC2 Security Group:

bash 
# Find the worker node Security Group ID
aws ec2 describe-instances \
  --filters "Name=private-ip-address,Values=192.168.7.124" \
  --query "Reservations[].Instances[].SecurityGroups" \
  --region ap-south-1

# Add inbound rule for the NodePort
aws ec2 authorize-security-group-ingress \
  --group-id <sg-xxxxxxxxx> \
  --protocol tcp \
  --port 32606 \
  --cidr 0.0.0.0/0 \
  --region ap-south-1
⚠ Without opening this port in the Security Group, the browser will show ERR_CONNECTION_TIMED_OUT — the connection simply drops. This is not an application error.

// Key Concepts

What You Learned

★
Ingress vs Service
Why Ingress exists and what problem it solves

Without Ingress, each app needs its own LoadBalancer Service = one ELB per app = expensive. Ingress consolidates all routing through one entry point.

Without IngressWith Ingress
3 apps = 3 LoadBalancers3 apps = 1 LoadBalancer (or NodePort)
3 different IPs/ports for users1 IP, paths differentiate apps
No URL-based routingFull path/host-based routing
TLS on each service separatelyTLS terminated once at Ingress
★
rewrite-target Annotation
Why the annotation prefix matters

When a user hits /pyapp, NGINX needs to forward just / to the backend — most apps don't know they're served under a subpath. rewrite-target: / strips the path prefix.

AnnotationResult
nginx.ingress.kubernetes.io/rewrite-target: /Correct — NGINX reads it
ingress.kubernetes.io/rewrite-target: /Wrong prefix — silently ignored
★
LoadBalancer vs NodePort on EKS Academy
When and why to use each
Service TypeReal AWS AccountAWS Academy
LoadBalancer Works — creates ELB Blocked — OperationNotPermitted
NodePort Works Works — use this
  • NodePort exposes a port in range 30000–32767 on every worker node
  • Traffic hits node-public-ip:nodeport and reaches the NGINX controller
  • NGINX then routes to the correct backend based on the URL path
  • Security Group must explicitly allow the NodePort — default blocks all
