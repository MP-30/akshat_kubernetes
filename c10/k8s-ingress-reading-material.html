Chapter 01

The Problem — Why Ingress Exists

Before we understand what Ingress is, we need to understand the problem it solves. Most technical concepts make much more sense once you feel the pain of not having them.

Imagine you are deploying three applications on a Kubernetes cluster — a user-facing website, an API, and an admin dashboard. Each of these needs to be accessible from the internet.

The naive solution in Kubernetes is to give each application a LoadBalancer Service. This works. Each service gets its own external IP address or DNS name, and traffic flows in. Problem solved, right?

Not quite. Here is what that looks like in the real world:

3 apps = 3 Load Balancers = 3 bills on AWS, 3 different IPs for users to remember

This approach has three serious problems:

Cost. Every LoadBalancer on AWS provisions an actual Elastic Load Balancer. Each one costs money — and the bill grows linearly with your number of applications. Ten microservices means ten ELBs.

Complexity. Your users need to remember three different IP addresses or DNS names. Your SSL certificates need to be configured on each LoadBalancer separately. Your rate limiting, authentication middleware, logging — all duplicated.

No smart routing. You can't say "requests to /api go here, requests to /admin go there" — each service is completely isolated.

The core insight: What if there was a single, smart entry point that received ALL incoming traffic and then decided — based on the URL — which application to forward it to? That is exactly what Ingress provides.

With Ingress, the picture becomes dramatically simpler:

Chapter 02

The Hotel Lobby Analogy

The best way to truly understand Ingress is through an analogy. And there is no better analogy than a large hotel.

Think of your Kubernetes cluster as a large hotel building. The hotel has many departments — a restaurant, a gym, a spa, room service, and hundreds of guest rooms. Guests arrive at one place: the front entrance. — The Hotel Analogy

Here is how every piece of the Ingress system maps to a hotel:

🚪
The Front Entrance + Driveway
All guests arrive here. There is only one entrance — one address you give people when they ask "where is the hotel?" In Kubernetes, this is your single IP address and port. One way in, for everyone.
= NodePort / LoadBalancer (the public IP:port)
📋
The Receptionist at the Lobby Desk
The receptionist greets every guest, checks what they want, reads their name/booking, and decides where to send them. "You booked the restaurant? Third floor. Gym? Basement. Room 402? Elevator on the right." The receptionist does not serve the food or operate the gym — they only route people to the right place.
= NGINX Ingress Controller (reads request, routes to right backend)
📖
The Hotel Rulebook at the Desk
The receptionist doesn't memorize rules — they follow a written rulebook. "If guest says 'restaurant', send to floor 3. If guest says 'room service', call extension 4. If guest shows a booking for a suite, send to the executive floor." The rulebook tells the receptionist exactly what to do for each type of request.
= The Ingress YAML file (the routing rules you define)
🍽️
The Individual Departments
The restaurant, gym, spa, and guest rooms all operate independently. They don't care how guests found them — they just serve whoever shows up. Each department is only reachable through internal corridors, not directly from the street.
= Your backend Services and Pods (pyapp-svc, petclinic-svc etc.)

Now consider two ways a hotel receptionist might route guests:

Path-Based Routing — "What do you want?"

Imagine all guests arrive at the same hotel address. The receptionist looks at what they are asking for — the purpose of their visit — and routes accordingly:

🏨Hotel Analogy — Path-BasedRouting by purpose

A guest walks in and says: "I'm here for the restaurant." — Receptionist: "Third floor, turn left."

Another guest: "I need the gym." — Receptionist: "Basement level, take the stairs."

Another: "I want my room." — Receptionist: "Room 402, elevator to floor 4."

All guests came through the same front door. The purpose they stated determines where they go.

In Kubernetes: All requests arrive at the same IP. The URL path (/restaurant, /gym, /api) determines which backend service handles the request.

Host-Based Routing — "Which hotel did you mean to visit?"

Now imagine a hospitality company owns three separate hotel brands — a budget hotel, a business hotel, and a luxury resort — all operating from the same physical building complex. Each has its own branded entrance sign and its own service style. Guests arrive at the same physical address but announce which brand they are checking into:

🏨Hotel Analogy — Host-BasedRouting by brand/identity

A guest walks in: "I have a booking at Budget Inn." — Receptionist routes to the budget wing.

Another: "I'm checking into Grand Business Suites." — Receptionist routes to the business wing.

Another: "Royal Luxury Resort, my name is Singh." — Receptionist routes to the luxury wing.

Same building, same entrance, but the brand name they state determines their entire experience.

In Kubernetes: All requests arrive at the same IP. The Host header (pyapp.example.com, petclinic.example.com) sent by the browser determines which backend service handles the request.
Chapter 03

Ingress vs Ingress Controller

This is one of the most common points of confusion for beginners. People apply their Ingress YAML, nothing works, and they don't understand why. The answer is almost always: the Ingress Controller was never installed.

Kubernetes has a clear separation between what you want and who does it. The Ingress object describes what you want. The Ingress Controller is the software that actually does it.

🗺️Analogy — Rules vs PersonThe policy and the enforcer

Think of a traffic rulebook at a large junction. The rulebook says: "Red cars turn left. Blue cars go straight. Green cars take the flyover." The rules are written down, clear, and correct.

But if there is no traffic cop standing at the junction reading and enforcing those rules, cars just pile up in chaos. The rulebook alone does nothing.

Ingress YAML = The traffic rulebook. It defines the rules. By itself, it is just a config file stored in Kubernetes — it does nothing to actual traffic.

Ingress Controller = The traffic cop. It reads the rulebook and physically routes packets to the correct destination. Without installing it, your Ingress rules are ignored.

There is another important distinction: Kubernetes ships with many controllers built-in (for Deployments, Services, ConfigMaps) but it does not ship with an Ingress Controller. You must install one yourself. The most common choices are:

ControllerWho Makes ItWhen to Use
NGINXKubernetes communityMost common, great for labs and production both
AWS ALB ControllerAmazonEKS production — integrates with AWS ALB natively
TraefikTraefik LabsGood auto-discovery, popular with Docker Swarm backgrounds
HAProxyHAProxy TechnologiesHigh performance, fine-grained control needed

In our lab we use the NGINX Ingress Controller installed from the official Kubernetes community manifest. Once installed, it runs as a pod inside your cluster in the ingress-nginx namespace and constantly watches for Ingress objects to process.

The ingressClassName field: Because a cluster can have multiple controllers installed simultaneously, the ingressClassName: nginx field in your Ingress YAML tells Kubernetes which controller should handle this particular Ingress object. Think of it as addressing your rulebook to the right traffic cop.
Chapter 04

Path-Based Routing — Routing by URL

Path-based routing is the simpler of the two routing types to understand. Every HTTP request has a URL. The URL has a path — the part after the domain name. For example, in http://example.com/api/users, the path is /api/users.

In path-based Ingress, you write rules that say: "if the path starts with X, send the request to service Y."

yaml — path-based ingress
spec:
  rules:
  - http:
      paths:
      - path: /pyapp        # requests to /pyapp go to pyapp-svc
        pathType: Prefix
        backend:
          service:
            name: pyapp-svc
            port: {number: 80}
      - path: /petclinic    # requests to /petclinic go to petclinic-svc
        pathType: Prefix
        backend:
          service:
            name: petclinic-svc
            port: {number: 80}

The rewrite-target Problem

Path-based routing introduces a subtle but important issue. When a user visits mysite.com/pyapp/home, NGINX forwards the full path including /pyapp to the backend. But the pyapp application was built to handle requests at / — it has no idea it's being served under /pyapp. So it gets a request for /pyapp/home and returns 404 because it only knows about /home.

📬Analogy — The Post OfficeAddress stripping

Imagine a post office that sorts mail. A letter arrives addressed to "Building A, Department Finance, John Smith." The post office routes it to Building A. But when they hand it to John Smith inside the building, they just hand him the letter — they don't say "here's your letter that came via Building A, Department Finance." The routing prefix is stripped. John just sees his letter.

rewrite-target: / does exactly this — it strips the routing prefix (/pyapp) before forwarding the request to the backend. The application receives a clean request at / and works normally.
Critical detail: The annotation must be nginx.ingress.kubernetes.io/rewrite-target — not ingress.kubernetes.io/rewrite-target. The NGINX controller only reads annotations with its own prefix. Using the wrong prefix means the annotation is silently ignored and the rewrite never happens.
Chapter 05

Host-Based Routing — Routing by Hostname

Every HTTP request carries a Host header. This is how the browser tells the server which website it's trying to reach. When you type pyapp.example.com into your browser, the browser performs a DNS lookup to find the IP address, connects to that IP, and then sends:

http — raw request headers
GET / HTTP/1.1
Host: pyapp.example.com
User-Agent: Mozilla/5.0 ...
Accept: text/html ...

Notice the Host: header. Even though the browser connected to the same IP address as petclinic.example.com, the Host header is different. This is the crucial information NGINX uses for host-based routing.

📞Analogy — The Call CentreSame number, different departments

A large company has one phone number — say 1800-COMPANY. When you call, an automated system answers and asks: "Which department would you like? Press 1 for Sales, 2 for Support, 3 for Billing."

Everyone dials the same number. But what you say or press immediately determines who actually talks to you. The routing decision happens at the entry point based on your stated intent.

Now imagine instead of pressing numbers, callers simply say the department name — "Sales" or "Support" or "Billing." The system hears the word and routes accordingly.

In host-based Ingress: Everyone connects to the same IP. But the Host header they send ("pyapp.example.com" or "petclinic.example.com") is the word they say. NGINX hears it and routes to the matching backend.

What Changes in the YAML

The only difference between path-based and host-based Ingress is the addition of a host: field in each rule. That single field changes the entire routing mechanism:

yaml — host-based ingress
spec:
  rules:
    # Without host: field = path-based (matches any hostname)
    # With host: field = host-based (only matches this hostname)

  - host: "pyapp.65.2.186.80.nip.io"   # ← this is the only addition
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: pyapp-svc
            port: {number: 80}

  - host: "petclinic.65.2.186.80.nip.io"
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: petclinic-svc
            port: {number: 80}

What is nip.io and Why Do We Use It?

For host-based routing to work, the hostname in your browser must resolve to the IP of your Ingress Controller. In production, you would create a proper DNS A record pointing pyapp.yourcompany.com to your load balancer IP.

In a lab, setting up a real DNS server is unnecessarily complex. nip.io solves this elegantly — it is a free public DNS service with a simple rule:

🌐How nip.io WorksMagic wildcard DNS

Any hostname of the form anything.IP_ADDRESS.nip.io automatically resolves to that IP address. No account, no configuration, no DNS record creation — it just works.

pyapp.65.2.186.80.nip.io → resolves to → 65.2.186.80

petclinic.65.2.186.80.nip.io → resolves to → 65.2.186.80

Both resolve to the same IP — but the hostname the browser sends in the Host header is different. That difference is exactly what NGINX uses to route to the right backend.

nip.io solves the DNS problem so you can practice host-based routing in any lab environment without needing a domain name or DNS server.
Chapter 06

How NGINX Makes the Routing Decision

Understanding the exact order in which NGINX evaluates rules will save you hours of debugging. There is a precise priority system.

🔍Analogy — The Airport Check-In CounterPriority-based sorting

At a busy airport, passengers approach a long check-in counter. The counter has sections labeled by airline and destination. A staff member at the entrance applies a strict priority system:

First: They check if the passenger has a booking for a specific airline — this is like the host: field. If they match, the passenger is directed to that airline's section.

Second: Within that section, they look at the destination city — this is like the path:. A passenger going to Delhi gets one queue, someone going to Mumbai gets another. More specific destinations win — a passenger with "Delhi, Terminal 2, Gate 14" goes to a more specific counter than one just saying "Delhi."

Third: If nobody recognizes the passenger at all, they go to a general information desk — this is the defaultBackend.

In precise technical terms, NGINX evaluates every incoming request as follows:

StepNGINX DoesWhat Wins
1Finds all rules where the host: field matches the request's Host headerExact match wins over wildcard match
2Among matching host rules, finds paths that match the request URLLonger/more specific path wins over shorter path
3If no rule matches at alldefaultBackend serves it, or 404 if none configured

A practical example of path specificity — imagine these three rules all under the same host:

yaml — path specificity example
paths:
- path: /           # matches everything — least specific
  backend: default-svc
- path: /api        # matches /api and /api/* — more specific
  backend: api-svc
- path: /api/v2     # matches /api/v2/* — most specific
  backend: api-v2-svc

A request to /api/v2/users will be handled by api-v2-svc — because /api/v2 is a longer, more specific match than /api which is more specific than /. The most specific path always wins.

Useful debugging trick: Use curl -v http://IP:NodePort -H "Host: pyapp.example.com" to manually set the Host header. This bypasses DNS entirely and lets you test if NGINX routing is working correctly, independent of whether DNS resolves properly.
Chapter 07

The NodePort Workaround on AWS Academy

When you install the NGINX Ingress Controller on EKS using the standard manifest, it creates a Service of type LoadBalancer. In a normal AWS account, this automatically provisions an Elastic Load Balancer and gives the controller a public IP.

AWS Academy accounts, however, restrict ELB creation. The service stays stuck showing EXTERNAL-IP <pending> forever, and the event log shows a clear error:

error message
OperationNotPermitted: This AWS account currently does not support
creating load balancers.
🏢Analogy — Building with No Main DoorUsing the service entrance instead

Imagine a hotel building where the main entrance was locked and the management won't open it (that's AWS Academy blocking LoadBalancer creation). The hotel still functions perfectly — all the departments, rooms, and services work fine. You just can't use the main entrance.

But there is a side door — the service entrance — that is always open. It's not as elegant (visitors have to remember a specific door number), but it leads to the same lobby, the same receptionist, and the same routing system.

NodePort is the service entrance. Instead of a clean DNS name from an ELB, visitors use the worker node's raw public IP and a specific port number (32606). Less elegant, but functionally identical — traffic still reaches the NGINX controller and all routing works exactly the same.

The fix is a simple one-line patch:

bash — patch service to NodePort
kubectl patch svc ingress-nginx-controller \
  -n ingress-nginx \
  -p '{"spec": {"type": "NodePort"}}'

After patching, Kubernetes assigns a port from the range 30000–32767 (for example, 32606). This port is opened on every worker node in the cluster — so hitting any worker node's public IP on that port reaches the NGINX controller.

Always include the NodePort in your URL. When browsers see a URL like http://pyapp.65.2.186.80.nip.io with no port, they default to port 80. Your NGINX controller is not on port 80 — it's on 32606 (or whatever was assigned). This is why the browser shows a timeout: the packet arrives at port 80, nothing is listening, the connection is refused or dropped. Always use http://pyapp.65.2.186.80.nip.io:32606.

One more thing: AWS Security Groups block all inbound traffic by default. After patching to NodePort, you must explicitly open that port in the EC2 worker node's Security Group, or the traffic never reaches the node at all — producing another timeout with no helpful error message.

Chapter 08

Everything Together — The Full Picture

Now that every concept is clear, let us walk through exactly what happens — from the moment you type a URL in your browser to the moment your application's response appears on screen.

Assume the URL is http://pyapp.65.2.186.80.nip.io:32606.

StepWhat HappensWhere
1Browser parses the URL. Sees hostname pyapp.65.2.186.80.nip.io, port 32606.Your browser
2Browser queries DNS for pyapp.65.2.186.80.nip.io. nip.io answers: 65.2.186.80.Public DNS (nip.io)
3Browser opens a TCP connection to 65.2.186.80:32606.Network
4EC2 Security Group checks: is port 32606 allowed inbound? If yes, packet reaches the node.AWS Security Group
5NodePort 32606 on the worker node receives the packet. Kubernetes routes it to the NGINX Ingress Controller pod.EKS cluster
6NGINX reads the HTTP request. Finds the Host: pyapp.65.2.186.80.nip.io header.NGINX pod
7NGINX checks its rules (loaded from your Ingress YAML). Finds matching rule: host = pyapp.* → backend = pyapp-svc.NGINX pod
8NGINX forwards the request to pyapp-svc (ClusterIP). Service load-balances to one of the pyapp pods.Kubernetes networking
9pyapp pod processes the request and sends a response back through the same chain.pyapp pod
10Your browser displays the page.Your browser

Notice that the backend services (pyapp-svc, petclinic-svc) are all ClusterIP type — they are not exposed to the internet at all. Only the NGINX Ingress Controller is externally accessible. All other services are purely internal, reachable only by NGINX. This is also a significant security benefit of using Ingress.

Key Takeaways
  • Ingress solves the problem of needing one LoadBalancer per application — a single entry point handles all traffic.
  • The Ingress object is just a YAML config file — it defines rules but does nothing by itself.
  • The Ingress Controller is the running software (a pod) that reads those rules and routes actual network traffic.
  • Path-based routing uses the URL path (/api, /admin) as the routing key. More specific paths always win.
  • Host-based routing uses the HTTP Host header as the routing key — one IP can serve dozens of distinct domains.
  • rewrite-target strips the path prefix before forwarding to the backend, so apps don't need to know they're behind a proxy.
  • nip.io provides free wildcard DNS for labs — any subdomain.IP.nip.io resolves to that IP, enabling host-based routing without real DNS setup.
  • NodePort is the workaround when LoadBalancer is unavailable — functionally identical, just accessed via node IP + port number instead of a clean DNS name.
