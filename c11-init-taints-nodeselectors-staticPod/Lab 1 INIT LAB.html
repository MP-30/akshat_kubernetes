Before You Start — What is an Init Container?

Read this first — takes 2 minutes

An init container is a special container that runs before your main app starts. Think of it like a security guard at the door — the main app only enters after the guard says "all clear".

Runs firstInit Container
→
Runs secondInit Container 2
→
Starts lastMain App ✅
Simple rules to remember:
✅ Init containers run one by one (not together)
✅ Each must finish successfully before the next starts
✅ Main app only starts after all inits finish
❌ If init fails → pod restarts and tries again
Status you'll see Meaning
Init:0/11 init container, still running (0 done)
Init:1/22 init containers, first one done
PodInitializingAll inits done, main container starting
RunningMain app is live ✅
Init:CrashLoopBackOffInit container failed ❌

Exercise 1 — Your First Init Container

Init writes a message → main app reads it

In this exercise, the init container writes a text file. The main app reads that file. They share data using a volume.

📁
A volume is like a shared folder between containers inside the same pod. Both the init container and main container can read/write to it.

Step 1 — Create the YAML file

ex1-init.yaml 
apiVersion: v1
kind: Pod
metadata:
  name: myfirst-init
  namespace: init-lab
spec:

  # ---- INIT CONTAINERS run FIRST ----
  initContainers:
  - name: write-message
    image: busybox:1.35
    command: ["sh", "-c", "echo 'Hello from init!' > /shared/msg.txt && echo 'Init done!'"]
    volumeMounts:
    - name: myvolume
      mountPath: /shared

  # ---- MAIN CONTAINER runs AFTER init ----
  containers:
  - name: read-message
    image: busybox:1.35
    command: ["sh", "-c", "echo 'Main started! Message is:' && cat /shared/msg.txt && sleep 3600"]
    volumeMounts:
    - name: myvolume
      mountPath: /shared

  # ---- Shared volume ----
  volumes:
  - name: myvolume
    emptyDir: {}

Step 2 — Apply and watch

terminal 
# Apply the pod
kubectl apply -f ex1-init.yaml

# Watch status (you will see Init:0/1 → Running)
kubectl get pod myfirst-init -w
# what you should see NAME READY STATUS RESTARTS myfirst-init 0/1 Init:0/1 0 myfirst-init 0/1 PodInitializing 0 myfirst-init 1/1 Running 0

Step 3 — Read the logs

terminal 
# See what the init container printed
kubectl logs myfirst-init -c write-message

# See what the main container printed
kubectl logs myfirst-init -c read-message
# write-message logs Init done! # read-message logs Main started! Message is: Hello from init!
✅
Exercise 1 done! You saw the init container write a file and the main app read it. The main app started only after init finished.

Exercise 2 — Wait for a Service

Init keeps checking until a service is ready

Real apps often need to wait for a database or another service to be ready. The init container loops until it can connect, then exits and lets the main app start.

Step 1 — Deploy a fake "database" service

fake-db.yaml 
apiVersion: v1
kind: Pod
metadata:
  name: fake-db
  namespace: init-lab
  labels:
    app: fake-db
spec:
  containers:
  - name: db
    image: nginx:alpine   # nginx acts as our fake service
    ports:
    - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: db-service
  namespace: init-lab
spec:
  selector:
    app: fake-db
  ports:
  - port: 80

Step 2 — Deploy the app that waits for db-service

ex2-wait.yaml 
apiVersion: v1
kind: Pod
metadata:
  name: myapp
  namespace: init-lab
spec:
  initContainers:
  - name: wait-for-db
    image: busybox:1.35
    command:
    - sh
    - -c
    - |
      echo "Checking if db-service is ready..."
      until nc -z db-service 80; do
        echo "  db not ready yet, waiting 3 seconds..."
        sleep 3
      done
      echo "db-service is ready! Starting main app now."

  containers:
  - name: main-app
    image: nginx:alpine

Step 3 — Apply in order and observe

terminal 
# First deploy the app (it will WAIT for db)
kubectl apply -f ex2-wait.yaml

# Watch it — stays stuck in Init:0/1
kubectl get pod myapp -w

# Open a second terminal and check init logs
kubectl logs myapp -c wait-for-db -f

# Now deploy the database (in another terminal)
kubectl apply -f fake-db.yaml

# Watch myapp move from Init:0/1 → Running
# init logs while waiting Checking if db-service is ready... db not ready yet, waiting 3 seconds... db not ready yet, waiting 3 seconds... db-service is ready! Starting main app now. # pod status NAME READY STATUS RESTARTS myapp 0/1 Init:0/1 0 <-- waiting myapp 1/1 Running 0 <-- db deployed, app starts
✅
Exercise 2 done! Your app waited for the database before starting — no manual coordination needed. This is a very common real-world pattern.

Exercise 3 — Debug a Failing Init Container

Break it intentionally, then fix it

When an init container fails, your pod gets stuck. Let's practice how to debug it.

Step 1 — Deploy a broken pod

ex3-broken.yaml 
apiVersion: v1
kind: Pod
metadata:
  name: broken-pod
  namespace: init-lab
spec:
  initContainers:
  - name: bad-init
    image: busybox:1.35
    command: ["sh", "-c", "echo 'Starting...' && cat /no/such/file.txt"]
    # ↑ This will FAIL because the file doesn't exist

  containers:
  - name: main-app
    image: nginx:alpine
terminal 
kubectl apply -f ex3-broken.yaml
kubectl get pod broken-pod -w
# pod will keep failing NAME READY STATUS RESTARTS broken-pod 0/1 Init:Error 0 broken-pod 0/1 Init:CrashLoopBackOff 1

Step 2 — Use these 3 commands to debug

  • 1
    describe the pod — shows Events section at the bottom
    kubectl describe pod broken-pod
  • 2
    check init container logs — see the actual error message
    kubectl logs broken-pod -c bad-init
  • 3
    check previous run logs — if it already restarted
    kubectl logs broken-pod -c bad-init --previous
# kubectl logs broken-pod -c bad-init Starting... cat: can't open '/no/such/file.txt': No such file or directory

Step 3 — Fix and redeploy

terminal 
# Delete the broken pod
kubectl delete pod broken-pod

# Apply the fixed version below
kubectl apply -f ex3-fixed.yaml
ex3-fixed.yaml 
apiVersion: v1
kind: Pod
metadata:
  name: fixed-pod
  namespace: init-lab
spec:
  initContainers:
  - name: good-init
    image: busybox:1.35
    command:
    - sh
    - -c
    - |
      echo "Starting..."
      # Fixed: create the file first, then read it
      echo "hello" > /tmp/myfile.txt
      cat /tmp/myfile.txt
      echo "Init done!"

  containers:
  - name: main-app
    image: nginx:alpine
✅
Exercise 3 done! You now know how to identify a failing init container and fix it using describe and logs.
