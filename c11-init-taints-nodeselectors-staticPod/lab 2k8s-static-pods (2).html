// 01 — Fundamentals

What is a Static Pod?

In Kubernetes, most pods are created by submitting a manifest to the API server, which then schedules them via the scheduler and tracks them in etcd. A Static Pod is different — it bypasses all of that.

A static pod is a pod defined by a YAML file placed directly on a node's filesystem, in a directory that the kubelet watches (typically /etc/kubernetes/manifests/). The kubelet reads that file and starts the pod directly, without ever asking the API server for permission.

💡

Key insight: The kubelet is the only component that actually runs containers. Everything else in Kubernetes — the scheduler, controller-manager, API server — just tells the kubelet what to do. Static pods let you talk to the kubelet directly, skipping the chain of command.

What happens to a static pod in the API server?

When a static pod starts, the kubelet automatically creates a mirror pod in the API server so you can see it with kubectl get pods. However, this mirror pod is read-only — you cannot delete or modify it via kubectl. The kubelet owns it.

// 02 — Analogy

The Night Security Guard

// Real-World Analogy

Imagine a large office building. During the day, a central security management system (the Kubernetes API server) coordinates all guards, assigns them shifts, and tracks their locations. Guards follow instructions from this system.


But at night, there's one special guard — the Night Watchman. He doesn't need the central system to tell him to show up. He has a hardcoded schedule written on a card in his pocket (the manifest file on disk). He arrives at his post automatically, and if he falls asleep (pod crashes), he wakes himself back up — no dispatcher needed.


The central system can see him on the monitor (mirror pod), but it cannot reassign him or tell him to leave. Only removing the card from his pocket (deleting the manifest file) makes him stop.

This is exactly how static pods work. The kubelet is the Night Watchman. The manifest file is the card in his pocket. The API server can observe but not control him.

Why does this matter?

This design solves a chicken-and-egg problem: Kubernetes components like the API server, etcd, and the scheduler are themselves run as containers. But to run containers, Kubernetes must already be running. Static pods break this paradox — they let critical system components start themselves before any orchestration layer exists.

// 03 — Internals

How Static Pods Work

The Manifest Directory

The kubelet's config file (typically /var/lib/kubelet/config.yaml) specifies where to watch for static pod manifests:

/var/lib/kubelet/config.yaml 
staticPodPath: /etc/kubernetes/manifests

Any .yaml or .json file placed in this directory is picked up by the kubelet automatically — no kubectl apply needed.

Lifecycle

  • File created in /etc/kubernetes/manifests/ → kubelet detects it within seconds via inotify and starts the pod.

  • Pod crashes → kubelet automatically restarts it with exponential back-off, just like any pod with restartPolicy: Always.

  • File modified → kubelet tears down the old pod and creates a new one with the updated spec.

  • File deleted → kubelet stops and removes the pod. The mirror pod disappears from the API server too.

  • Node reboots → On startup, the kubelet scans the directory and re-creates all static pods before connecting to the API server.

⚠️

No rescheduling: Static pods are bound to one specific node. If that node goes down, the pod dies and nothing reschedules it elsewhere. That's the key trade-off.

// 05 — Case Study

Real-World Case Study

🏗️
kubeadm: Bootstrapping a Production Cluster
How every managed Kubernetes cluster (EKS, GKE, AKS) starts its control plane

When kubeadm init runs to create a new Kubernetes cluster, it faces the classic bootstrap paradox: the API server needs to run to schedule pods, but it needs to run as a pod itself.

The solution: static pods. kubeadm writes four manifest files into /etc/kubernetes/manifests/ on the master node:

$ ls /etc/kubernetes/manifests/ 
etcd.yaml
kube-apiserver.yaml
kube-controller-manager.yaml
kube-scheduler.yaml

The kubelet starts up, scans this directory, and launches all four control plane components as static pods — before any Kubernetes orchestration exists. Within seconds, the cluster bootstraps itself from nothing.

This is why on a control plane node you'll see pods like kube-apiserver-master-node — the suffix is the node name, a telltale sign it's a static pod (the mirror pod naming convention).

✅

Resilience in action: If the API server pod crashes, the kubelet restarts it automatically — even if the API server itself is down. This is why control planes are so resilient. The watchdog is the process it's watching.

🌐
Edge Computing — Air-Gapped Nodes
IoT gateways running critical services without cluster connectivity

A logistics company runs Kubernetes on 500+ edge nodes inside warehouses. Network partitions happen frequently. The network monitoring agent on each node must keep running even when the node is completely disconnected from the cluster.

By deploying the monitoring agent as a static pod, it continues to collect metrics, write to local storage, and self-heal even with zero connectivity to the API server. When the network recovers, the data is synced back. A DaemonSet pod would simply be in a disconnected state with no guarantee of local restart.

// 06 — Hands-On

Lab: Create Your First Static Pod

🧪 Lab — Static Pod Lifecycle

Prerequisites: A running Kubernetes cluster with node access (minikube, kind, or a real node). You need SSH access to the node where you want the static pod to live.

Step 1 — Find the static pod directory

node shell 
# Check kubelet config for staticPodPath
cat /var/lib/kubelet/config.yaml | grep staticPodPath

# Expected output:
staticPodPath: /etc/kubernetes/manifests

# If not set, check the kubelet service flags:
ps aux | grep kubelet | grep static

Step 2 — Write the static pod manifest

/etc/kubernetes/manifests/my-static-nginx.yaml 
apiVersion: v1
kind: Pod
metadata:
  name: my-static-nginx
  labels:
    app: static-demo
spec:
  containers:
  - name: nginx
    image: nginx:1.25
    ports:
    - containerPort: 80
    resources:
      requests:
        cpu: "100m"
        memory: "64Mi"
      limits:
        cpu: "200m"
        memory: "128Mi"
Write the file to disk 
sudo vim /etc/kubernetes/manifests/my-static-nginx.yaml
# paste the YAML above and save

# Or use heredoc:
sudo tee /etc/kubernetes/manifests/my-static-nginx.yaml <<'EOF'
... (paste yaml) ...
EOF

Step 3 — Observe the pod appear

kubectl (from control plane) 
# Watch pods in real-time (within ~5 seconds the pod should appear)
kubectl get pods -A -w

NAMESPACE   NAME                           READY   STATUS    RESTARTS   AGE
default     my-static-nginx-<node-name>    1/1     Running   0          8s

# Note: the pod name has the NODE NAME appended — that's the mirror pod!

# Describe it — notice "Controlled By: Node/<name>" not a ReplicaSet
kubectl describe pod my-static-nginx-<node-name>

Step 4 — Try to delete it via kubectl (it won't work!)

The immovable pod 
kubectl delete pod my-static-nginx-<node-name>
pod "my-static-nginx-<node-name>" deleted

# Wait 3 seconds, then list again...
kubectl get pods
my-static-nginx-<node-name>    1/1     Running   0          3s

# 😄 It's back! kubectl deleted the mirror pod, but kubelet recreated it immediately.

Step 5 — Simulate a crash / self-healing

Kill the container, watch it heal 
# On the node, find and kill the nginx container
sudo crictl ps | grep my-static-nginx
abc123def456   nginx:1.25   Running   my-static-nginx   ...

sudo crictl stop abc123def456

# Back on control plane — watch the RESTARTS counter increment
kubectl get pods -w
my-static-nginx-node   0/1     CrashLoopBackOff   1          4s
my-static-nginx-node   1/1     Running            1          7s
# Restarted automatically by kubelet!

Step 6 — Modify the manifest (rolling update)

Change image version 
# On the node, edit the file
sudo sed -i 's/nginx:1.25/nginx:1.26/' \
  /etc/kubernetes/manifests/my-static-nginx.yaml

# kubelet detects the change and recreates the pod
kubectl get pods -w
my-static-nginx-node   1/1   Terminating   0   2m
my-static-nginx-node   1/1   Running       0   5s

Step 7 — Delete the static pod (properly)

The ONLY way to stop a static pod 
# On the node, remove the manifest file
sudo rm /etc/kubernetes/manifests/my-static-nginx.yaml

# Pod disappears from the cluster within seconds
kubectl get pods
No resources found in default namespace.
# ✓ Gone for good
🎉

Lab complete! You've seen: static pod creation, mirror pod naming, kubectl delete resistance, self-healing, live modification, and proper deletion — all the key behaviors of static pods.
