Node Selector Lab

Schedule prod pods on prod-node · Schedule test pods on test-node

⏱ 20 minutes
🟢 Beginner
2 nodes needed
kubectl
💡

What We Are Doing

Understand the goal before touching anything

Your cluster has two nodes. You want to make sure the right pods always land on the right node — production pods on the production node, test pods on the test node. We will do this using Node Selector — a simple label-based rule inside the pod YAML.

prod-node
env=prod
🟢 prod pods run here
↔
test-node
env=test
🟠 test pods run here
ℹ️
Node Selector works by attaching a label to a node, then telling the pod "only schedule me on a node that has this label." It is the simplest scheduling rule in Kubernetes.
1

Get Your Node Names

Find the actual names of your two nodes

Run this command to see your nodes. Note down the names — you will use them in the next step.

terminal 
kubectl get nodes
# example output — your names will be different NAME STATUS ROLES AGE prod-node Ready worker 2d test-node Ready worker 2d
📝
In this lab we call the nodes prod-node and test-node. Replace these with your actual node names in every command.
2

Label the Nodes

Attach env=prod and env=test labels to each node

A label is just a key=value tag. We label each node so pods can target it using nodeSelector.

terminal 
# Label the production node
kubectl label node prod-node env=prod

# Label the test node
kubectl label node test-node env=test

# Verify the labels were applied
kubectl get nodes --show-labels | grep env
# expected output prod-node Ready ... env=prod,... test-node Ready ... env=test,...
✅
Both nodes are now labelled. The nodes themselves are not changed in any way — we just added a tag. Pods can now use nodeSelector to pick which node they land on.
3

Deploy the Production Pod

This pod will only run on prod-node

Create a file called prod-pod.yaml with the content below. Notice the nodeSelector line — it tells Kubernetes to only schedule this pod on a node where env=prod.

prod-pod.yaml 
apiVersion: v1
kind: Pod
metadata:
  name: prod-pod
  labels:
    env: prod
spec:

  # Only schedule on nodes with label env=prod
  nodeSelector:
    env: prod

  containers:
  - name: prod-app
    image: nginx:alpine
    ports:
    - containerPort: 80
terminal 
kubectl apply -f prod-pod.yaml

# Check which node it landed on (-o wide shows the NODE column)
kubectl get pod prod-pod -o wide
# expected output NAME READY STATUS NODE prod-pod 1/1 Running prod-node ← landed on the right node ✅
✅
The NODE column shows prod-node. The nodeSelector worked — Kubernetes picked prod-node because it has the env=prod label.
4

Deploy the Test Pod

This pod will only run on test-node

Same idea — but this time nodeSelector targets env=test.

test-pod.yaml 
apiVersion: v1
kind: Pod
metadata:
  name: test-pod
  labels:
    env: test
spec:

  # Only schedule on nodes with label env=test
  nodeSelector:
    env: test

  containers:
  - name: test-app
    image: nginx:alpine
    ports:
    - containerPort: 80
terminal 
kubectl apply -f test-pod.yaml

# Check which node it landed on
kubectl get pod test-pod -o wide
# expected output NAME READY STATUS NODE test-pod 1/1 Running test-node ← landed on the right node ✅
✅
The NODE column shows test-node. Both pods are now on their correct nodes.
5

View Both Pods Together

Confirm each pod is on the correct node

terminal 
kubectl get pods -o wide
NAME READY STATUS NODE prod-pod 1/1 Running prod-node test-pod 1/1 Running test-node
🎉
Perfect! prod-pod is on prod-node. test-pod is on test-node. Node Selector is working exactly as expected.
6

Bonus — What if No nodeSelector?

A pod with no selector can land on any node

If a pod has no nodeSelector, Kubernetes picks any available node. Try it:

no-selector-pod.yaml 
apiVersion: v1
kind: Pod
metadata:
  name: random-pod
spec:
  # No nodeSelector — Kubernetes picks any node
  containers:
  - name: app
    image: nginx:alpine
terminal 
kubectl apply -f no-selector-pod.yaml
kubectl get pod random-pod -o wide

# It can land on EITHER node — no preference set
⚠️
Without nodeSelector, the scheduler picks any node. This is why we use nodeSelector — to make placement predictable and intentional.
