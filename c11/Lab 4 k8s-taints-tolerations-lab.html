Taints & Tolerations Lab

Keep prod pods on prod-node · Keep test pods on test-node

⏱ 20 minutes
🟢 Beginner
2 nodes needed
kubectl
💡

What We Are Doing

Understand the goal before touching anything

Your cluster has two nodes — one for production, one for testing. We want a hard rule: prod pods only on prod-node, test pods only on test-node.

We do this using Taints on the nodes and Tolerations on the pods.

prod-node
Taint: env=prod:NoSchedule
🟢 Only prod pods allowed
↔
test-node
Taint: env=test:NoSchedule
🟠 Only test pods allowed
🚫 Taint (on the Node)
A taint is like a "No Entry" sign on a node. It repels all pods that don't have the matching permission (toleration).
✅ Toleration (on the Pod)
A toleration is the pod's "permission pass". A pod with the right toleration is allowed past the taint and can land on that node.
ℹ️
Simple rule: A pod can only land on a tainted node if the pod's toleration matches the node's taint. No toleration = pod stays away.
1

Get Your Node Names

Find the actual names of your two nodes

Run this first. Note down the exact names of your two nodes — you will use them in every command below.

terminal
kubectl get nodes
# example — your names will be different NAME STATUS ROLES AGE prod-node Ready worker 2d test-node Ready worker 2d
📝
Replace prod-node and test-node with your actual node names in every command that follows.
2

Taint Both Nodes

Put a "No Entry" sign on each node

A taint has three parts: key=value:effect

We use NoSchedule as the effect — this means no pod can land on this node unless it has the matching toleration.

terminal
# Taint the production node
kubectl taint node prod-node env=prod:NoSchedule

# Taint the test node
kubectl taint node test-node env=test:NoSchedule

# Verify — check taints are applied
kubectl describe node prod-node | grep Taint
kubectl describe node test-node | grep Taint
# expected output Taints: env=prod:NoSchedule Taints: env=test:NoSchedule
✅
Both nodes are now tainted. From this point on, no pod can land on either node unless it has the correct toleration.
3

Deploy the Production Pod

This pod tolerates the prod-node taint — lands on prod-node

This pod has a toleration that matches the taint on prod-node. That is its permission to land there. It also has a nodeSelector so it only targets prod-node and not any other node.

prod-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: prod-pod
  labels:
    env: prod
spec:

  # This toleration = permission to land on prod-node
  tolerations:
  - key: "env"
    operator: "Equal"
    value: "prod"
    effect: "NoSchedule"

  # This ensures the pod targets prod-node specifically
  nodeSelector:
    kubernetes.io/hostname: prod-node

  containers:
  - name: prod-app
    image: nginx:alpine
ℹ️
Replace prod-node in the nodeSelector with your actual prod node name.
terminal
kubectl apply -f prod-pod.yaml

# Check which node it landed on
kubectl get pod prod-pod -o wide
# expected output NAME READY STATUS NODE prod-pod 1/1 Running prod-node ✅
✅
The NODE column shows prod-node. The toleration gave the pod permission, and nodeSelector pointed it to the right place.

5

View Both Pods Together

Confirm each pod is on the correct node

6

Bonus — Prove the Boundary Works

A pod with no toleration gets stuck — cannot land anywhere

Deploy a pod with no toleration at all. Since both nodes are tainted, no node will accept it. It will stay Pending forever.

plain-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: plain-pod
spec:
  # No tolerations — will be blocked by both taints
  containers:
  - name: app
    image: nginx:alpine
terminal
kubectl apply -f plain-pod.yaml
kubectl get pod plain-pod -o wide

# See WHY it is stuck
kubectl describe pod plain-pod | grep -A5 Events
# pod is stuck — no node will accept it NAME READY STATUS NODE plain-pod 0/1 Pending <none> # describe Events shows the reason 0/2 nodes are available: 1 node had untolerated taint {env: prod} 1 node had untolerated taint {env: test}
⚠️
This proves the boundary is real. No pod without the correct toleration can sneak onto either node. This is exactly how production teams protect dedicated node pools.

Quick Reference

Commands and cleanup

CommandWhat it does
kubectl get nodesList all nodes
kubectl taint node <name> key=val:NoScheduleAdd a taint to a node
kubectl describe node <name> | grep TaintCheck taints on a node
kubectl get pods -o wideShow pods with their node
kubectl describe pod <name>See events and scheduling info
kubectl taint node <name> key=val:NoSchedule-Remove a taint (trailing - removes it)

Cleanup — run after the lab:

terminal
# Delete all pods
kubectl delete pod prod-pod test-pod plain-pod

# Remove taints from both nodes (trailing - removes the taint)
kubectl taint node prod-node env=prod:NoSchedule-
kubectl taint node test-node env=test:NoSchedule-

# Verify nodes are clean
kubectl describe node prod-node | grep Taint
kubectl describe node test-node | grep Taint
