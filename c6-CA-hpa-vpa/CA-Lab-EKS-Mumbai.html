CONCEPT
How Cluster Autoscaler Works

Cluster Autoscaler watches for pods that cannot be scheduled (Pending state) and nodes that are consistently underutilised. It adds or removes EC2 nodes accordingly — automatically.

Scale-Up Flow

Pod created
→
No space on nodes
Pending state
→
CA detects Pending
within 30s
→
CA calls AWS ASG
increase desired
→
New EC2 node joins
~2-3 minutes

Scale-Down Flow

Pods deleted
→
Node becomes idle
<50% utilised
→
CA waits 10 min
cool-down
→
CA drains node
moves pods
→
EC2 terminated
cost saved

SituationCA ActionTiming
Pod stuck in PendingAdds new EC2 node via ASGDetects in 30s, node ready in ~3 min
Node idle <50% for 10 minDrains pods, terminates node10 min cool-down then ~2 min drain
All pods scheduled, nodes healthyDoes nothingPolls every 10 seconds
💡
CA requires resource requests on pods
CA calculates node utilisation using resources.requests on pods. If pods have no requests, utilisation is always 0% and CA will never scale down. Always set CPU and memory requests on every container.

STEP 1 — CRITICAL FIRST
IAM Permissions (Fix the #1 Error)
🚨
Most Common Error — Do This BEFORE deploying CA
Without this step, CA crashes immediately with AccessDenied: autoscaling:DescribeAutoScalingGroups. The node's IAM role does not have permission to call AWS autoscaling APIs by default.

The Error You Will See (if you skip this step)

CA CRASH LOG
E aws_manager.go:128] Failed to regenerate ASG cache: AccessDenied:
  User: arn:aws:sts::386757865964:assumed-role/eksctl-my-eks-cluster-nodegroup-st-NodeInstanceRole-wgOYaObz1jHu/i-0385...
  is not authorized to perform: autoscaling:DescribeAutoScalingGroups
  because no identity-based policy allows the action

Step 1A — Find Your Node Instance Role Name

The role name is visible in the error above, or run this command:

BASH
# Get the node instance role name from your nodegroup
aws eks describe-nodegroup \
  --cluster-name my-eks-cluster \
  --nodegroup-name standard-workers \
  --region ap-south-1 \
  --query 'nodegroup.nodeRole' \
  --output text
💡
How to find your nodegroup name
aws eks list-nodegroups --cluster-name my-eks-cluster --region ap-south-1

Step 1B — Attach the Required Policy

Replace the role name with your actual role from Step 1A:

BASH
# Attach AutoScaling permissions to the node IAM role
aws iam attach-role-policy \
  --region ap-south-1 \
  --role-name eksctl-my-eks-cluster-nodegroup-st-NodeInstanceRole-wgOYaObz1jHu \
  --policy-arn arn:aws:iam::aws:policy/AutoScalingFullAccess

What permissions does CA need?

AWS API PermissionWhy CA Needs It
autoscaling:DescribeAutoScalingGroupsDiscover which ASGs exist and their current size/state
autoscaling:SetDesiredCapacityTell the ASG to add more nodes (scale up)
autoscaling:TerminateInstanceInAutoScalingGroupRemove a specific node (scale down)
ec2:DescribeLaunchTemplateVersionsKnow the instance type to simulate scheduling decisions
✅
Verification
aws iam list-attached-role-policies --role-name <your-role> --region ap-south-1 — should now show AutoScalingFullAccess in the list.

STEP 3
Find and Tag Your Auto Scaling Group

CA discovers which ASG to manage using two specific AWS tags. First find the ASG name, then add the tags.

3A — Find ASG Name

BASH
# List all ASGs in Mumbai
aws autoscaling describe-auto-scaling-groups \
  --region ap-south-1 \
  --query 'AutoScalingGroups[*].AutoScalingGroupName' \
  --output text

# Save the output as a variable (replace the name below)
ASG_NAME=eks-standard-workers-4ecf26bb-3d18-100c-e84a-806e585bc33c
echo $ASG_NAME

3B — Tag the ASG

🚨
Wrap each tag in double quotes — this is mandatory
Without quotes you get: Expected: '=' received: ' '. Each tag entry must be a single quoted string.
BASH
# Tag the ASG — double quotes around EACH tag (critical!)
aws autoscaling create-or-update-tags \
  --region ap-south-1 \
  --tags \
  "ResourceId=${ASG_NAME},ResourceType=auto-scaling-group,Key=k8s.io/cluster-autoscaler/enabled,Value=true,PropagateAtLaunch=false" \
  "ResourceId=${ASG_NAME},ResourceType=auto-scaling-group,Key=k8s.io/cluster-autoscaler/${CLUSTER_NAME},Value=owned,PropagateAtLaunch=false"

Why these specific tags?

Tag KeyValuePurpose
k8s.io/cluster-autoscaler/enabledtrueTells CA this ASG is eligible to be managed
k8s.io/cluster-autoscaler/my-eks-clusterownedScopes CA to only manage ASGs for this specific cluster — prevents CA from touching other clusters' ASGs
PropagateAtLaunch=false—Don't copy this tag onto individual EC2 instances, only on the ASG itself
BASH — VERIFY
# Confirm tags were applied
aws autoscaling describe-auto-scaling-groups \
  --region ap-south-1 \
  --auto-scaling-group-names ${ASG_NAME} \
  --query 'AutoScalingGroups[0].Tags[*].Key' \
  --output text
# Should show both tag keys

STEP 5
Configure Region and Cluster Name

The downloaded manifest has us-east-1 and <YOUR CLUSTER NAME> as placeholders. Fix both using kubectl patch — no editor needed.

Fix 1 — Set Correct Region

BASH
kubectl -n kube-system set env deployment/cluster-autoscaler \
  AWS_REGION=ap-south-1

Fix 2 — Find the Cluster Name Argument Index

BASH
# Print all CA arguments with their index (0-based)
kubectl -n kube-system get deployment cluster-autoscaler \
  -o jsonpath='{.spec.template.spec.containers[0].command}' | tr ',' '\n'

# Output will look like:
# [0] "./cluster-autoscaler"
# [1] "--v=4"
# [2] "--stderrthreshold=info"
# [3] "--cloud-provider=aws"
# [4] "--skip-nodes-with-local-storage=false"
# [5] "--expander=least-waste"
# [6] "--node-group-auto-discovery=...<YOUR CLUSTER NAME>"  ← this is index 6

Fix 3 — Patch Cluster Name (no editor needed)

BASH
# Replace index 6 with your actual cluster name
kubectl -n kube-system patch deployment cluster-autoscaler \
  --type=json \
  -p='[{"op":"replace","path":"/spec/template/spec/containers/0/command/6","value":"--node-group-auto-discovery=asg:tag=k8s.io/cluster-autoscaler/enabled,k8s.io/cluster-autoscaler/my-eks-cluster"}]'
⚠️
If you get "index out of range"
Count the lines from the jsonpath output above. Find which line contains <YOUR CLUSTER NAME> and replace /6 with the correct index number.

Verify Both Fixes

BASH
kubectl -n kube-system get deployment cluster-autoscaler \
  -o jsonpath='{.spec.template.spec.containers[0].command}' | tr ',' '\n'
# Should now show: my-eks-cluster (not YOUR CLUSTER NAME)
# And env should show: AWS_REGION=ap-south-1

CA Deployment Manifest — Key Fields Explained

--cloud-provider=aws
--skip-nodes-with-local-storage=false
--expander=least-waste
--node-group-auto-discovery=asg:tag=k8s.io/cluster-autoscaler/enabled,k8s.io/cluster-autoscaler/my-eks-cluster
ArgumentValueWhat It Means
--cloud-provider aws Tells CA which cloud it is running on. It uses AWS-specific APIs (ASG, EC2) to add/remove nodes.
--skip-nodes-with-local-storage false By default CA will NOT remove a node that has pods using local storage (emptyDir, hostPath). Setting false means CA CAN remove such nodes — useful in labs but be careful in production.
--expander least-waste When CA needs to pick which node group to scale up, it uses this strategy. least-waste picks the group that results in the least unused CPU/memory — most cost-efficient. Other options: random, most-pods, priority.
--node-group-auto-discovery asg:tag=... Tells CA how to find which ASGs to manage — by looking for ASGs that have both discovery tags. The format is asg:tag=key1,key2. CA calls DescribeAutoScalingGroups to find matching ASGs.
--v=4 4 Log verbosity level. 4 = detailed logs including scaling decisions. Useful for debugging. Set to 1 or 2 in production to reduce log noise.

STEP 7
Trigger Scale-Up

Deploy 10 pods each requesting 500m CPU and 512Mi memory. Your 2-node cluster cannot fit all of them — some enter Pending state, which triggers CA to add a node.

ca-test.yaml
cat <<EOF | kubectl apply -f -
apiVersion: apps/v1
kind: Deployment
metadata:
  name: ca-test
spec:
  replicas: 10
  selector:
    matchLabels:
      app: ca-test
  template:
    metadata:
      labels:
        app: ca-test
    spec:
      containers:
      - name: app
        image: registry.k8s.io/pause:3.1
        resources:
          requests:
            cpu: 500m
            memory: 512Mi
EOF

Manifest Fields Explained

FieldValueExplanation
replicas 10 Create 10 identical pod copies. With 2 t3.medium nodes (~2 vCPU, 4Gi each), only ~6 pods can fit. The remaining ~4 pods will be Pending — this is what triggers CA.
image: pause:3.1 — The pause container does absolutely nothing — it just sleeps. Perfect for testing scheduling and scaling without actually doing any work. Tiny image, starts instantly.
resources.requests.cpu 500m 500m = 0.5 vCPU. This is what Kubernetes reserves on the node for this pod. t3.medium has 2 vCPU so it can fit about 3 pods on CPU alone (minus system overhead).
resources.requests.memory 512Mi 512 Mebibytes reserved per pod. t3.medium has 4Gi (~4096Mi) so can fit about 6 pods on memory alone (minus system). The combination of CPU + memory determines real capacity.
selector.matchLabels app: ca-test The Deployment uses this label to find which pods it owns. Must match the labels in the pod template. If these differ, the Deployment cannot manage the pods.

BASH — WATCH SCALE-UP
# Terminal 1 — watch pods (some will be Pending initially)
kubectl get pods -l app=ca-test

# Terminal 2 — watch nodes (new node appears in ~3 minutes)
kubectl get nodes --watch

# Terminal 3 — watch CA logs for scaling decision
kubectl -n kube-system logs deployment/cluster-autoscaler -f | grep -i scale

# Look for:
# "Scale up: adding 1 node to group"
# "Successfully added node"

STEP 8
Trigger Scale-Down
BASH
# Delete the deployment — nodes become idle
kubectl delete deployment ca-test

# CA will wait the cool-down period (~10 min default) then remove idle nodes
kubectl get nodes --watch

# Watch scale-down in CA logs
kubectl -n kube-system logs deployment/cluster-autoscaler -f | grep -i 'scale down'

# Look for:
# "Scale down: removing node"
# "node removed by cluster autoscaler"
💡
Why does scale-down take 10 minutes?
Scale-down is intentionally slow to avoid thrashing — rapidly adding and removing nodes if traffic is spiky. The 10-minute window (--scale-down-unneeded-time) ensures a node is consistently idle before being removed. Scale-up is fast because adding capacity is always safe; scale-down removes capacity so CA is more cautious.

INTERVIEW PREP
Common CA Interview Questions
Q
What triggers CA to scale up?

CA scales up when one or more pods are in Pending state because they cannot be scheduled due to insufficient CPU or memory on existing nodes. CA runs a simulation: it checks if adding a node from any configured node group would allow the pending pod to be scheduled. If yes, it increases the ASG desired count by 1 and waits for the new node to join.

Q
What prevents CA from removing a node?

Several things block scale-down: pods with no controller (bare pods), pods with local storage (emptyDir), pods annotated with cluster-autoscaler.kubernetes.io/safe-to-evict: "false", pods blocked by a PodDisruptionBudget, and nodes with custom taints. Also, a node must be under the utilisation threshold (50% default) for the entire --scale-down-unneeded-time window (10 min default).

Q
CA vs Karpenter — key differences?
FeatureCluster AutoscalerKarpenter
Speed3-5 min (via ASG)<60 sec (direct EC2 API)
Instance selectionFixed per node groupAutomatic best-fit
ConsolidationManual/limitedAutomatic bin-packing
SetupSimplerMore IAM setup required
Q
What is an expander in CA?

An expander is the strategy CA uses to decide which node group to scale up when multiple groups could fit the pending pod. Options: least-waste (fewest unused resources — recommended), most-pods (schedules most pods), random (random choice), priority (user-defined ranking). least-waste is best for cost efficiency.
