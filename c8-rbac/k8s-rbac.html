What is RBAC?

RBAC lets you control which users or services can perform which actions on which resources — without granting blanket cluster-admin access.

Subject

Who is making the request. Can be a human user, a service account, or a group.

kind: User | ServiceAccount | Group

Role / ClusterRole

A set of rules — each rule specifies which verbs are allowed on which resources.

RoleBinding

Glues a subject to a role within a namespace (or cluster-wide for ClusterRoleBinding).

RBAC Objects

Four Kubernetes API objects make up the entire RBAC system.

ServiceAccount / User / Group

Subjects are identities. ServiceAccounts (system:serviceaccount:ns:name) live inside Kubernetes. Users and Groups come from your external identity provider (OIDC, certificates, etc.).

Role — namespaced

A Role contains rules valid only within one namespace. You define which API groups, resources, and verbs (get, list, watch, create, update, patch, delete) are permitted.

ClusterRole — cluster-wide

A ClusterRole works the same as Role but applies across all namespaces — or for non-namespaced resources like nodes, persistentvolumes, and clusterroles themselves.

RoleBinding

Binds one or more subjects to a Role or ClusterRole within a specific namespace. A ClusterRole referenced by a RoleBinding is scoped to that namespace only.

ClusterRoleBinding

Binds subjects to a ClusterRole cluster-wide. Use sparingly — it grants permissions across every namespace.

YAML Examples

A complete example: read-only pod access in production namespace for a CI service account.

role.yaml 
apiVersion: rbac.authorization.k8s.io/v1
kind:       Role
metadata:
  name:      pod-reader
  namespace: production
rules:
  - apiGroups: [""]
    resources: [pods, pods/log]
    verbs:
      - get
      - list
      - watch
rolebinding.yaml 
apiVersion: rbac.authorization.k8s.io/v1
kind:       RoleBinding
metadata:
  name:      read-pods
  namespace: production
subjects:
  - kind:      ServiceAccount
    name:      ci-runner
    namespace: production
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind:     Role
  name:     pod-reader
serviceaccount.yaml 
apiVersion: v1
kind:       ServiceAccount
metadata:
  name:      ci-runner
  namespace: production

# Kubernetes auto-mounts a token
# for this SA in every Pod that
# specifies serviceAccountName:
#   ci-runner
kubectl verify 
# Can ci-runner list pods?
$ kubectl auth can-i list pods \
    --as=system:serviceaccount:
    production:ci-runner \
    -n production
yes

# Can it delete pods?
$ kubectl auth can-i delete pods \
    --as=system:serviceaccount:
    production:ci-runner \
    -n production
no

Namespace vs Cluster Scope

Choosing the right object type determines whether permissions are sandboxed or global.

✓ Best practice

Grant the minimum scope needed. Prefer Role + RoleBinding over ClusterRole + ClusterRoleBinding whenever possible.

⚠ Watch out

A ClusterRoleBinding cannot be scoped down later without deleting it. Audit bindings regularly with kubectl get clusterrolebindings.
